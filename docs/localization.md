# Localization

Reference for localizing this app via String Catalogs (`.xcstrings`). The catalog lives at `Kios/Resources/Localizable.xcstrings`. The extension has its own at `KiosControls/Localizable.xcstrings`.

The build emits string metadata automatically ‚Äî `SWIFT_EMIT_LOC_STRINGS: YES` and `LOCALIZATION_EXPORT_SUPPORTED: YES` (see `project.yml:25-26`). Every localizable string in source becomes an entry in the catalog on next build.

Supported regions today: `en` (source) + `de`. See `project.yml:2-4`.

## String Catalogs vs the old way

| | Old (.strings / .stringsdict) | New (.xcstrings) |
|---|---|---|
| File format | Plain text | JSON, edited via Xcode UI |
| Plurals | Separate .stringsdict file | Same catalog |
| Device variations | Verbose | Inline |
| Extraction | Manual via genstrings | Automatic at build |
| Stale-key detection | None | Marked `extractionState: stale` |

New code ‚Üí `.xcstrings`. Don't add `.strings` or `.stringsdict` files.

## How extraction works

`SWIFT_EMIT_LOC_STRINGS: YES` makes the compiler emit metadata for every `LocalizedStringKey`, `LocalizedStringResource`, and `String(localized:)` it sees. Xcode merges that metadata into your catalog on build:

- New string in source ‚Üí added to catalog as `state: "new"`.
- String removed from source ‚Üí catalog entry marked `extractionState: "stale"`.
- Translations are preserved across both.

You almost never need to add entries manually. Write the string in code, build, the catalog updates.

## In SwiftUI

### `Text` takes a `LocalizedStringKey` by default

```swift
Text("Library")                       // ‚Üê key "Library", auto-localized
```

The literal becomes the key. Translations go in the catalog.

### Pass `LocalizedStringKey` explicitly when you need to

When a custom view wants to take a localized string parameter:

```swift
// Kios/Views/Editorial/EditorialComponents.swift
struct EditorialNavBar<TitleContent: View, Trailing: View>: View {
    init(title: LocalizedStringKey, eyebrow: String? = nil, ...) { ... }
}
```

Take `LocalizedStringKey`, not `String`. That way the caller can pass a literal `"Library"` and it gets extracted.

### When you have a runtime `String`

```swift
Text(verbatim: book.title)            // not localized ‚Äî display raw
Text(book.title)                      // ‚Üê compile error in iOS 17+ for String
```

`Text(verbatim:)` is the right call for user-provided strings (book titles, author names). They aren't translatable, so don't pretend.

### Interpolation

```swift
Text("\(percent)% complete")
Text("\(book.title) by \(author)")
```

The macro extracts `"%lld%% complete"` and `"%@ by %@"`. Translators see the placeholders and know how to reorder for grammar:

```json
"%@ of %@" : {
    "localizations" : {
        "de" : { "stringUnit" : { "value" : "%1$@ von %2$@" } }
    }
}
```

German uses `%1$@`/`%2$@` to reorder ‚Äî the catalog supports positional args.

## Outside SwiftUI

For Swift code that isn't a `View` body (services, App Intents, errors):

```swift
let resource = LocalizedStringResource("Settings")
let s = String(localized: "Library")
let formatted = String(localized: "\(count) books")
```

This codebase uses `LocalizedStringResource` for `AppIntent` titles:

```swift
// Kios/Intents/OpenMostRecentBookIntent.swift
static let title: LocalizedStringResource = "Open Most Recent Book"
```

And for error messages exposed to the system:

```swift
enum OpenMostRecentBookError: Error, CustomLocalizedStringResourceConvertible {
    var localizedStringResource: LocalizedStringResource {
        switch self {
        case .noBook: "No book to open yet."
        }
    }
}
```

`LocalizedStringResource` is `Sendable`, `Codable`, and can be passed across actor boundaries (unlike a raw `String` "key" with metadata).

## Comments

Add a comment to every non-obvious key. Translators see the catalog without context:

```swift
Text("Open", comment: "Verb. Button label for opening a book.")
```

Why this matters: "Open" is one English word but maps to different words in German depending on whether it's an adjective ("offen") or verb ("√∂ffnen").

In `.xcstrings` JSON the comment becomes:

```json
"Open" : {
    "comment" : "Verb. Button label for opening a book."
}
```

Some auto-extracted comments are generated from accessibility labels ‚Äî see the catalog entry for `"%@, %@"` with `"isCommentAutoGenerated": true`.

## Pluralization

```swift
Text("\(count) books")
```

Extracts as `"%lld books"`. In the catalog, mark it as a plural rule and add variations:

```json
"%lld books" : {
    "variations" : {
        "plural" : {
            "one" : { "stringUnit" : { "value" : "1 book" } },
            "other" : { "stringUnit" : { "value" : "%lld books" } }
        }
    }
}
```

Edit this through Xcode's catalog UI ‚Äî right-click the entry ‚Üí **Vary by Plural**. The UI shows the categories required for each locale (English needs `one` / `other`; Russian needs `one` / `few` / `many` / `other`).

## Device variations

Same key, different copy on iPad/iPhone:

```json
"Library" : {
    "variations" : {
        "device" : {
            "iphone" : { "stringUnit" : { "value" : "Library" } },
            "ipad"   : { "stringUnit" : { "value" : "Your Library" } }
        }
    }
}
```

Not heavily used in this codebase (iPhone-only ‚Äî see `TARGETED_DEVICE_FAMILY: "1"` in `project.yml:51`).

## Formatters

Numbers, dates, durations: use a formatter, not a hand-written interpolation. Format styles localize automatically:

```swift
Text(date, format: .dateTime.month().day())
Text(count, format: .number)
Text(duration, format: .units(allowed: [.minutes, .seconds]))
Text(percent, format: .percent)
```

See `Kios/Views/Home/StatsFormatters.swift` for the project's wrapper functions.

For currency, percent, units, prefer `FormatStyle` over `NumberFormatter` ‚Äî the format-style API is composable and value-typed.

## Catalog hygiene

### Stale entries

When you delete or rename a string in source, the catalog marks the old key `extractionState: stale`:

```json
"%@ of %@" : {
    "extractionState" : "stale",
    "localizations" : { ... }
}
```

Stale entries don't cause build errors. Periodically open the catalog in Xcode and:

- Delete confirmed-dead entries (translations are gone ‚Äî back them up first if needed).
- Re-bind if the same string moved to a different surface.

### "Should translate" flag

Some strings are display-only and shouldn't be translated:

```json
"%@ ¬∑ %lld" : { "shouldTranslate" : false }
"¬∑" : { "shouldTranslate" : false }
```

Set this from the Xcode UI (the **State** column). Useful for separators, formatting glue, debug strings.

### Source language

`sourceLanguage: "en"` (top of catalog). Source language entries don't need to be "translated" ‚Äî the source string *is* the value.

## Common pitfalls

- **`String(localized:)` with interpolation captures variable values into the key.** `String(localized: "Hello \(name)")` extracts `"Hello %@"`, not `"Hello Alice"`. The macro handles this correctly because the format string is a literal. If you build the format string at runtime, extraction breaks.
- **`Text(someString)`** where `someString` is a runtime `String`. Compiler accepts this (treats as `LocalizedStringKey`) and tries to look up `someString` as a key. Use `Text(verbatim:)` for user-provided strings.
- **`%d` vs `%lld`.** Swift's `Int` is 64-bit on iOS ‚Äî extracted strings use `%lld`. Don't manually edit to `%d`; the catalog and source must match.
- **Reusing one key for two strings.** "Open" used as both a button label and a status word will share one translation ‚Äî and German wants different words. Use distinct keys: `"action.open"` vs `"status.open"`. Or use the `comment:` parameter to differentiate (the comment becomes part of the key's identity in some catalog versions).

## Patterns from this codebase

### Editorial components take `LocalizedStringKey`

```swift
// Kios/Views/Editorial/EditorialComponents.swift
struct EditorialNavBarStringTitle: View {
    let title: LocalizedStringKey
    var body: some View { Text(title)... }
}
```

The custom design system passes `LocalizedStringKey` through every layer. Call sites use literals (`"Library"`, `"Settings"`), and extraction Just Works.

### Stable accessibility labels too

```swift
struct EditorialNavIconButton: View {
    var accessibilityLabel: LocalizedStringKey
    ...
}
```

Accessibility labels are user-facing text. They get localized the same way.

### `LocalizedStringResource` in App Intents

App Intent titles, descriptions, and error messages all use `LocalizedStringResource`. The system reads them from the catalog at runtime ‚Äî which is essential for Shortcuts surfaces that render outside your app.
